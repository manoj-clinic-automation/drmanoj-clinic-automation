## §§1–7 — Consolidated base (through Session 73, from v1.28)

*(Reproduced verbatim from the v1.28 consolidated master.)*

## 1. The data spine — VERIFIED at its headwater (Docterz)

Docterz is source of truth for patient identity + diagnosis; the tracker DERIVES the CSVs; the plan-tool
and vitals tool only READ Docterz-derived data and NEVER write back. **`Patient_UID` is generated by
Docterz** (native column, copied through by the tracker) and is the single backend join/storage key
across vitals, plan, visits, recordings, calls. It is a **backend field, invisible at the front of
Docterz** — reception/owner see only Clinic ID + name + mobile at the visit (D129). `Clinic_Specific_Id`
is the human lookup handle only (D128).

```
Docterz (headwater; mints Patient_UID, backend-only)
   │  export → tracker
   ▼
Docterz-derived CSVs  (patient_master, patient_diagnosis, visit_ledger)   ← READ-ONLY to the tools
   │  join on Patient_UID
   ▼
Hosted plan-tool + vitals tool  ── writes its OWN files ──►  vitals_ledger.csv
                                                             plan_ledger.csv
                                                             plan_archive/  (PDFs, D132)
```

One-writer-per-file law holds: the measurement lives once in `vitals_ledger`; the plan points at it
(`Vitals_ID_Used`); the PDF is the frozen output, pointed at from `plan_ledger`. Nothing duplicated to drift.

## 2. Operating decision — host plan-tool + vitals TOGETHER, once the backend works

The Track-1 target is to **build the whole thing (plan-tool + vitals + backend write-path) and host it
at ONE time.** Offline artifacts (v24 lookup, v25 vitals, v26 plan-record) are staging steps — the
"final online version" is locked only after the backend actually writes ledgers + PDFs end-to-end. This
tightens the definition of "done" (hosting = done) without changing the locked build order. The hosted
tool becomes a single hub-and-spoke spoke that owns its write files and only reads the Docterz CSVs.

## 3. The two ledgers (schemas locked; KB §67.3)

- **`vitals_ledger.csv`** — append-only, one row per capture; the single vitals writer computes
  BMI/category/waist:height for EVERY row regardless of which front door captured it. 20 columns (S67).
- **`plan_ledger.csv`** — choices-only, one row per plan generation; references vitals via
  `Vitals_ID_Used`. **14 columns after S73** (added `Plan_PDF_Patient` + `Plan_PDF_Physio`, D134).

Two front doors, one vitals writer (D127): the staff BP-only page (measurements only, no on-screen calc)
and the vitals section inside the plan-tool (owner). Both write through the same one writer, so the
ledger is always complete (D131).

## 4. New layer — printout PDF archive (D132/D133/D134)

A third tool-owned write target: **`plan_archive/`** — a patient-tagged store of the actual PDFs handed
to the patient.
- **Both sheets** (patient + physio) archived per visit — the frozen record of what was given.
- **Tagged on the Patient_UID spine** — `plan_archive/<year>/<Patient_UID>/<Plan_Date>_<Plan_ID>_{patient|physio}.pdf`;
  new patients → `pending/<Clinic_Id>_<mobile>/…`, stitched on reconciliation (D131).
- **VPS is home (D133)** — reliable local writes, no dependency on Drive at print time; Drive mirror
  deferred (reuse OAuth-as-owner pattern D36 if ever wanted).
- **Server-side generation** — PDFs produced by the hosted server, not the browser; archiving is a
  hosting-stage capability. Offline the tool previews the paths only.
- **Cost negligible** — ~100 MB/year at current load (<10 printed plans/day). Yearly folders → easy archive-off.

## 5. New-patient path + reconciliation (D130/D131)

Same-day new patients (not yet in the tracker) are captured on Clinic ID + name + mobile only (UID blank,
"pending sync"). A light hosted-stage reconciliation job later stitches the real Docterz UID (match on
Clinic ID + mobile), and moves any `pending/` PDFs into the real UID folder. Schema already supports it
(UID nullable; Clinic ID + name always present).

## §12 — Session 122: Stage-3 AI judge + Drive-token incident

**The audit pipeline reached Stage 3.** The four-stage Staff Call Audit design (record → transcribe
→ AI verdict → dashboard) now has its third stage live-capable. `call_verdict.py` is the **AI judge**:
it reads Sarvam transcripts, and — shown ONLY the transcript text + direction + duration, never the
staff's claimed outcome or any patient/agent identifier (the blind-judge rule, D62) — independently
classifies each call into the clinic's OWN live dashboard outcome vocabulary, sets six safety flags
(postop · complaint · urgent · surgery · clinical · conduct), and quotes the deciding phrase as
evidence. Python then compares the AI's blind verdict against the staff's claim → Match / Mismatch /
Partial / Unclear. Verdicts land in a doctor-only `Call_Verdicts` tab; v1 is classify-and-flag only —
no auto-accept, no downstream actions. This is the accountability layer the whole audit project was
built toward: it verifies claimed outcomes against what the recording actually contains, without ever
exposing patient identity to the AI.

**Architecture principles honoured:** one-writer-per-table (`call_verdict.py` alone owns
`Call_Verdicts`); walled-off responsibility (it never edits any other table and triggers no actions);
secrets in `.env` only; calibration-first (human reviews everything before any automation). The
provider/model sits behind one `.env` line, so the judge is swappable without a rebuild.

**Incident — the shared-token single point of failure.** Stages 1, 2, and 3 all authenticate to
Drive with one owner-OAuth token. This session exposed that the token had been minted while the OAuth
app was in **Testing** status, which Google expires after 7 days — so the token silently died and
would have taken down BOTH nightly jobs (with Stage-1 recording loss being permanent, given
MyOperator's 24-hour links). Fixed permanently by publishing the app to **In production**, and
immediately by re-minting the token. The lesson for the architecture: **the recordings pipeline has a
single shared credential whose failure is silent and high-consequence — arming the timer-freshness
checker is now a priority, not a nicety.**

**Open architectural gap (next):** the claim-match join. Pairing a call to the staff's claimed
outcome by mobile + a ±45-minute time window proved too fragile against real staff behaviour (batch
filing hours after the call). The blind-verdict half is sound; the join half needs a stronger key
(mobile + agent-extension + same-day). Until then the judge reports "No claim logged" rather than a
false mismatch — safe, but incomplete.

**Decision:** **D149** (Stage-3 AI judge) — see Master KB §122 for the full rule.

## §13 — Session 123: Stage-3 claim-match join redesigned; verdict layer proven

**Stage 3 became trustworthy.** The AI judge already worked; this session fixed the join that pairs
each call to the staff's filed outcome, and proved the whole layer on real data.

**The architectural correction (D150).** The old design assumed a claim is filed right after the
call, and matched within ±45 minutes. Reality: staff file outcomes in **morning batches**, so a
claim's timestamp is *filing-time*, not *call-time*. A tight clock window can never work. The join
now keys on the patient's **phone number** over a **whole-day forward window** (call → next-morning
batch), pairs multiple calls to one number in call-time order, and stamps each row with a **Match
Confidence** (`unique`/`ordered`/`none`) so the doctor knows how far to trust each verdict.

**Judge-once, fill-later (D151).** The costly step (the AI) runs exactly once per call. When the
staff claim arrives later, only the claim/verdict cells are updated — the doctor's own columns are
never touched, and re-runs cost nothing. This is the property that makes a future through-the-day
cadence affordable and lets the AI verdict + safety flags reach the doctor the *same day*, with the
staff-vs-AI diff filling itself in whenever the outcome is filed.

**Enriched, self-contained rows (D152).** Each verdict row now carries everything the doctor needs on
one line — full number, recording link (joined from the Stage-1 `Call_Recordings` tab by the shared
Join Key), match confidence, and a name/Clinic-ID fallback so even an unmatched call has an identity.

**What the data showed (D153).** On 06-Jul, 16 of 22 outgoing calls with a filed claim matched (73%),
with 5 honest mismatches and 3 safety flags surfaced. Incoming calls carry no staff outcome by
workflow, so "No claim logged" is correct there — not a gap.

**Architecture principles honoured:** one-writer-per-table (`call_verdict.py` still solely owns
`Call_Verdicts`); the blind-judge rule intact (the enriched fields are OUTPUT to the doctor-only
sheet, never sent to the AI); walled-off responsibility (it reads `Call_Recordings` but writes only
its own tab); manual/idempotent re-runs as the safety net.

**Next (D154, design-locked).** A **Verdict Analysis Layer** — a daily, read-only, one-patient-per-
screen-vertical view that segregates verdicts by scenario (Mismatch · AI-logged-staff-didn't ·
Unclear · Matches-collapsed) for fast doctor review — sits on top of this proven data as a separate
read-only surface, keeping one-writer-per-table intact.

**Decisions:** **D150–D153** (see Master KB §123 for the exact rules); **D154** design-locked.

## §10 — Session 102: staff call-sheet de-duplication (Track-1-adjacent, PC-side)

## §14 — Session 124: the review surface, and taking verification off the critical path

### 14.1 The layer that was missing
Stage 3 produced a 35-column horizontal sheet nobody could read. **`verdict_review.py` (D155)** turns it
into a reading surface: one patient per screen, segregated by scenario, the full transcript beneath each
card with the AI's evidence highlighted where it can be located — and, when it can't, a card that says so
rather than highlighting the wrong line.

Two new tables, each with exactly one writer, and `Call_Verdicts` never touched:
`verdict_review.py` → `Verdict_Review` (ephemeral, rolling, PHI, redrawn each run) and
`Doctor_Verdicts` (durable, append-only ground truth). **The ephemeral/durable split is the design.**
The rolling window makes the PHI surface self-expiring — a transcript leaves the tab as its call ages
out, which satisfies the locked 90-day purge rule without the purge job knowing the tab exists. The
durable ledger carries the doctor's adjudications, and it is the substrate for both the voice-bot KB and
the autonomous judge.

**The maturation mechanism, stated plainly:** every refereed card writes one row of ground truth. Once a
given outcome code has enough refereed examples and the AI agrees with the doctor above a threshold, that
code auto-accepts and stops reaching his desk. Codes below threshold keep coming. **The doctor's workload
falls as trust rises, code by code, and he can watch it fall.** Nothing about this is speculative; the
ledger exists. What does not yet exist is a single refereed row.

### 14.2 The principle that cost two clinic days to learn
The duration gate (D77/D82) was a **synchronous blocker** built on an **asynchronous, third-party,
best-effort** signal. On 06–08 Jul a silent webhook 403 meant no `Call_Durations` row could ever arrive,
so the gate never unlocked, so no staff member could record what any patient had said. Two clinic days of
outcome data, lost to a verification mechanism.

**D156 states the principle: no verification mechanism may stand between a staff member and recording
what a patient said.** Filing is the staff's job. Verifying is the system's job. They must not share a
critical path. Concretely: when a call *cannot be measured* the outcome unlocks behind an "unverified"
banner; when a call *is measured as not-connected* it still blocks. The gate keeps its real purpose and
loses its power to stop the clinic.

This generalises. The follow-up push, the Drive token, the AI key, the WABA relay — **every dependency
needs a degraded mode in which the humans keep working, and an alarm that reaches the doctor before the
clinic opens.** Stability is not the absence of breakage; it is failing open everywhere and alarming
loudly everywhere.

### 14.3 The inversion the staff UI needs (designed, not built)
Eleven outcome codes conflate two questions: *what happened* and *what happens next*. Only the second is
something a receptionist knows at hang-up and can be held to. On 06-Jul, **15 of 36 outgoing calls (42%)
carried no outcome at all**, and 4 of the 20 that did disagreed with the AI — every disagreement between
adjacent meanings (`coming` vs `on_medication`). That is not carelessness; it is an over-specified
vocabulary being filed from memory hours after the call.

**Staff answer "what next" in four or five choices. The machine answers "what happened" from the
transcript. The doctor adjudicates the disagreements.** Voicemail stops being a missing code and becomes
"couldn't reach". Batch filing stops when the question costs one tap at hang-up. And staff are never
shown the AI — corrective action reaches them from the doctor, about the patient, never from a machine
that is itself wrong one call in five.

### 14.4 Verification, and the limits of the written record
Two figures in the Session-123 record and one verification claim in the Session-94 record did not survive
contact with the live data (D157, D158, D159). A fix closed on the strength of a single immediate call was
dead seven minutes later. **The standing rule is now: one real call AND a re-check ≥1 hour later, same
clinic day.** And **D160**: the live Apps Script export is the canonical dashboard, because half of it had
never been committed and two diagnoses this session were made against a stale copy and were wrong.

**Decisions:** **D155** (Verdict Analysis Layer built), **D156** (gate fails open; verification off the
critical path), **D157** / **D158** (corrections to the S123 record; the claim-join defect remains open),
**D159** (the 403 recurrence), **D160** (live Apps Script is canonical). See Master KB §124.

## §15 — Sessions 125–127: the shared-secret architecture, and who does the checking

Three sessions on one fault produced two architectural changes and one change in method. The fault — `CALLHOOK_SECRET_MISMATCH_403` — cost four clinic days across two windows. Neither window was caused by carelessness.

### The architectural change: a shared secret is now allowed to disagree with itself

Before S125, `CALLHOOK_SECRET` existed in two places — the MyOperator panel and `/root/wa/.env` — with **nothing checking that they agreed**, and a receiver that refused mismatches **silently**. Change one end and the other is wrong, invisibly, until a receptionist notices stuck tiles three days later.

Three properties turned an ordinary fault into an outage, and all three are now closed:

| Property | Closure |
|---|---|
| A secret lives in two places, unchecked | **Dual-key gate (D162).** The receiver accepts `CALLHOOK_SECRET` **or** `CALLHOOK_SECRET_PREV`, constant-time. The two ends may disagree indefinitely. The disagreement surfaces as a WARN line naming the key in use. |
| A change takes effect at a moment no human selects | `gunicorn -w 1` with no `--preload` reads `.env` **once, at import**. A corruption sleeps until a worker respawns. **Now: restarts happen at a moment we choose, with a `cmp`-validated rollback in hand (D168).** |
| A door refuses people without saying so | **Refusals are written down before they are refused (D163)** — `call_hook_rejects/YYYY-MM-DD.jsonl`, metadata only, throttled. Absence of a raw log can no longer mean either *"nobody called"* or *"we rejected everybody."* |

**Generalises to every shared secret in the estate:** `WA_APPROVE_KEY`, `FU_UPLOAD_SECRET`, the WABA token, the service-account key. *Any secret read once at import in a single-worker process is a mine with an unknown fuse.*

### The second architectural change: the human is not the verifier

Session 127's rotation was executed by `rotate_callhook.sh`, not by a doctor reading forty exit codes. This was the owner's own diagnosis — *"again i am doing lot of un ending cmd job"* — and it is now a design rule (**D171**).

The insight is not about convenience. **A script that refuses is strictly safer than a human who approves.** `grep -c` exits 1 on a legitimate zero count; a human reading `exit=1` must decide whether it is the good kind, forty times, at 23:30, on a live clinical system. That is not a check. That is a coin toss dressed as diligence.

The guards the human was performing are now written into the tool: eleven of them, self-deleting candidate on failure, `cmp`-validated rollback point revalidated **at the instant before** the destructive operation. The human types one word.

**Standing rule: any procedure walked by hand twice should be walked by a script the third time.**

### The change in method: the record loses to the disk

Across S125–S127, **eight recorded facts failed against direct observation** — a byte count repeated over five sessions (`30,750` → `30,749`), a `sed` that never ran, a lost newline that never existed, a rollback point that rolled back to the wrong version, a version number in a title, a decision number (`D161`) that was never minted, a `§12` that did not contain what three sessions of documents said it contained, and — twice in one hour on 09-Jul — a count predicted from memory of a file rather than derived from it (**D172**).

Not one of these was found by reasoning. Every one was found by opening the file.

This is now the project's dominant failure mode, and it has a shape: *a plausible claim, written confidently, surviving because nobody re-opened the artefact.* Three countermeasures are in force — `UNKNOWN` is a legitimate entry (**D166**); expected values are derived, never predicted (**D172**); and every canonical document carries an **end-of-file assertion** so truncation cannot be silent. As of KB v1.49, all of them do.

## §17 — Session 131: a design finished, and four findings left standing

The session was told at the outset that nothing would be built. Nothing was. **The product is a design and eleven decisions.**

**F-8, anatomised.** `esc()` (L685) escapes `& < > "` and not `'`; `jsq()` (L729) escapes `\ '` and not `"`. L912 pushes `JSON.stringify` output through `jsq`; L923 pastes it into a `"`-delimited `onclick`; the attribute closes at the first `"` and **the handler is never installed**. The blast radius is **any `Patient_Master` match** — a bare UID breaks it too — not merely `known`.

**The two tile mechanisms.** Nothing in the fifteen-file project has ever written `Callbacks_Today.Staff Status`. Every `setValue`/`setValues` was searched. **The only thing that has ever cleared an incoming tile is a person typing into the spreadsheet by hand** — so a repaired button would have logged without clearing. Meanwhile `saveIncomingOutcome` **already computes `settle`/`escalate`/`retry`**, already writes it, and the client already renders it. The machine is built; nobody consumes its output.

**The loop already exists.** `sendBackToStaff` writes `SENT_BACK`; `getFollowups` (L938) re-surfaces the row as a staff tile under *"Sent back by doctor"* and auto-clears when staff file a newer outcome. `getEscalations` already attaches recording and transcript. **The owner's AI-review proposal is a second row source into an eighty-session-old loop.**

**The design.** The judge proposes; the doctor disposes; the staff act. Phase 1 moves no tile by machine — and the doctor's click *is* the referee decision that fills `Doctor_Verdicts`, so the calibration ledger builds itself out of a day's ordinary work. Conduct is never a number about a person: *a wrong tile-bounce costs one phone call; a wrong conduct score costs a person their standing with their employer.*

**What the design refuses to guess.** `script_not_followed` and `no_closing` are fully specified and **do not work**, because the clinic has no written script and no definition of a closing. Status `UNKNOWN` (D166/D199). *If there is no script, that absence is the finding, and the first fix is a script, not a judge.*

**The owner's impression — "conduct is good, script is strictly not followed, closing is weakest" — is recorded as a hypothesis, not a fact.** It has D190's exact shape. The training pack's first job is to test it.
